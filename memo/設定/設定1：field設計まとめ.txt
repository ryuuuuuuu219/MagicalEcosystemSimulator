field 設計まとめ

## 位置づけ

このファイルは field / 環境場の詳細正本。
集約入口は `設計集約/field・環境場設計.md` とする。

field は、個体行動・戦闘・魔法・群知性・支配種の土台になる。
個体は field を読み、戦闘と魔法は field を変化させ、UI は field を観測する。

## 基本分類

### scalar field

- threat score
- heat
- mana
- 火 / 氷が参照する heat
- 空間が参照する mana

値だけを持つ場。
AI は重みや閾値として読み、UI は濃度や等高線として表示する。
雷の scalar field は設計しない。

### vector field

- 低脅威方向
- heat gradient
- mana flow
- 風属性の三次元ベクトル場

方向を持つ場。
移動補正、誘導、回避、戦闘カメラの可視化に使う。

### tensor field

- 方向ごとの抵抗
- 属性流路
- 空間魔法の歪み
- 魔力場の異方性

方向ごとに意味が変わる場。
初期段階では debug 可視化と設計入口に留める。

## threat map

用途:
- 捕食者や高脅威地点の記録。
- 攻撃発生時の threat pulse。
- 低脅威方向探索。
- 回避、離脱、経路選好の補助入力。

現状:
- `threatmap_calc` は存在する。
- `scoreCellSize = 2f`
- `lowThreatProbeDistance = 3f`
- `lowThreatDirectionSamples = 12`
- threat score の加算、減衰、低脅威方向探索は実装済み。
- `ThreatMapsGenerator` で debug 可視化ができる。
- 攻撃時 threat pulse は実装済み。

未完:
- 草食・肉食の主移動ロジックへの全面統合。
- 警告行動、待ち伏せ、縄張り侵入を threat 発生源へ含める設計。
- 相ごとに field を分けるか、共通 field を重みで読むかの決定。

実装へ渡す時:
- `GetThreatScore`
- `GetLowThreatDirection`
- `EvaluateBestDirection`
- `AddThreatPulse`

を再利用し、別の threat 生成系を増やさない。

## heat field

用途:
- 環境熱量。
- 火属性 / 氷属性との接続。
- 危険温度からの回避。
- 将来の熱ダメージ、鈍足、属性耐性。

現状:
- `HeatFieldManager` が存在する。
- `gridResolution = 96`
- heat は代謝と死骸分解で場へ加算される。
- heat はグリッド上で拡散、減衰する。
- debug 描画がある。

未完:
- `SampleHeat` を動物の移動判断へ接続すること。
- 熱ダメージ、鈍足、属性耐性。
- 恒星由来の一様入熱。
- 「危険温度からの回避」だけにするか、「適温誘引」まで含めるか。

設計基準:
- heat は環境に存在する scalar field。
- 炭素保存則には含めない。
- 個体熱量 `Qbody` と環境熱量 `Qenv` を分けて考える。
- 拡散近傍は 4 近傍を初期基準とする。

## mana field

用途:
- 魔素濃度。
- 魔法発動コスト。
- 魔法補正。
- 支配種進化。
- 空間魔法の痕跡。

現状:
- 未実装。
- 設計上は拡散する scalar field。

基本式:

```text
Δmana = diffusionRate * (neighborAverage - self)
mana_next = mana + Δmana
```

設計:
- 魔法使用で消費する。
- 死骸分解で加算する。
- 転移では発動地点と転移先を結ぶ線分上へ分散加算する。
- 減衰はしない。
- 世代更新で 0 に初期化する。

初期実装候補:
- `ManaFieldManager`
- `SampleMana`
- `AddMana`
- `ConsumeMana`
- `GetManaGradient`

## 属性場

用途:
- 火 / 氷 / 風 / 空間の影響を field として扱う。
- 雷は field として扱わず、着弾・連鎖・短時間 pulse の event として扱う。

接続:
- 火: heat 加算、燃焼、熱ダメージ。
- 氷: heat 低下、鈍足。
- 雷: scalar field は設計しない。着弾イベント、連鎖イベント、短時間 pulse。
- 風: 三次元 vector field、移動補正、ノックバック、流れ。
- 空間: mana field、転移、歪み、抵抗。

未決定:
- 風の三次元 vector field の解像度、更新周期、可視化方法。
- 属性効果を個体状態へ直接入れるか、field 経由で入れるか。

## テンソル場

用途:
- 魔力流の方向差。
- 空間魔法の歪み。
- 地形や属性による方向別抵抗。
- 戦闘可視化。

扱い:
- 初期段階では実装本体ではなく、debug / UI 表示の設計入口にする。
- scalar / vector 表示が安定してから、tensor 表示へ広げる。

## UI / 可視化

最初の入口は `1.戦闘可視化強化`。

- 指定個体の周辺 field 値。
- 指定個体から見た低脅威方向。
- heat gradient。
- mana 濃度。
- 攻撃 pulse。
- wind vector。
- 雷の着弾・連鎖 event。

UI は field を更新しない。
field manager が値を持ち、UI は読むだけにする。

## 実装上の優先順位

1. 既存 threat map / heat field の重複実装を避ける。
2. AI が `GetThreatScore` / `GetLowThreatDirection` / `SampleHeat` を読める形にする。
3. mana field の最小 manager を決める。
4. scalar field の表示 API をそろえる。
5. 風の三次元 vector field の表示と実装を検討する。
6. tensor field の表示と実装を後段で検討する。
