現状メモから読み取れる属性決定方針

1. 属性は単なる見た目ではなく、field / 環境場への接続で意味が決まる。
- 火と氷は heat field と接続する。
- 空間は mana field と接続する。
- 風は三次元 vector field として扱う。
- 雷の scalar field は設計しない。雷は着弾イベント、連鎖イベント、短時間 pulse として扱う。

2. 属性ごとの基本性能差がある。
- 火: 単体寄り、高ダメージ、高め精度、遅め弾速、継続ダメージ、延焼、heat 加算。
- 氷: 狭め AoE、やや低ダメージ、高精度、火と同程度の弾速、棘生成、heat 低下、鈍足。
- 雷: 近傍連鎖、やや高ダメージ、最速、低精度、瞬間攻撃、次弾誘導、短時間イベント。
- 風: 広め AoE、低ダメージ、速い弾速、中精度、ノックバック、強制移動、推力。
- 空間: スタック脱出、位置補正、転移、歪み。支配種到達後能力として扱う。

3. 属性適性は魔法拡張ゲノムで扱う案がある。
追加候補として以下が明記されている。
- elementFireBias
- elementIceBias
- elementThunderBias
- elementWindBias
- manaAffinity
- manaControl
- spellUnlockRate
- antiMagicResponse
- castRiskTolerance
- cooldownDiscipline

このため、属性決定は「完全ランダム」ではなく、個体ごとの属性 bias と解放状態を重みとして使うのが自然。

4. 属性の解放状態は個体ごとに持つ想定がある。
必要な補助データとして「属性ごとの解放状態」が明記されている。
また、魔法属性にはオプションのスキルツリーが存在し、解放状況も遺伝するとされている。

5. 行動インデックスが属性・魔法選択に関わる。
メモでは、各魔法のフレームごとの使用確率とインデックスを優先度として扱う案がある。
使用数によってアンロックし、使用数によって遺伝子に補正する。
一定時間で各行動の確率をわずかに補正し、確率の大きい順を優先度とする。

6. 魔法攻撃は近接攻撃と同じ攻撃候補テーブルへ並べる。
攻撃設計では、近接 / 突進 / 噛みつき / 魔法を最終的に同じ攻撃候補テーブルへ並べる。
魔法拡張後は magicUsePriority を、近接、追跡、回避より魔法を優先する重みとして扱う。

7. 被弾による属性覚醒は通常アンロック・突然変異とは別枠。
被弾時に確率で、受けた属性に対抗しうる属性へ覚醒する。
これは肉体や耐性ではなく、攻撃側の魔法行動だけを覚醒対象にする。

相性回転:
- 火 < 氷
- 氷 < 雷
- 雷 < 風
- 風 < 火

つまり覚醒先は以下。
- 火を受けたら氷へ覚醒しうる。
- 氷を受けたら雷へ覚醒しうる。
- 雷を受けたら風へ覚醒しうる。
- 風を受けたら火へ覚醒しうる。

複数属性から被弾した場合、複数属性へ覚醒しうる。
ただし覚醒先は常に相性回転上の上位属性に限定する。

8. 覚醒で直接解放される技能は限定される。
覚醒で直接解放される技能は、専用の迎撃能力付き魔法に限定する案。
候補:
- 迎撃能力付き近接攻撃
- ノックバック強めの近接攻撃

覚醒後、そこから確率でより上位の飛翔体などを習得していく。
このため、被弾覚醒で即座に飛翔体属性を得るのではなく、まず短期的に試行されやすい迎撃系ノードを開ける設計が自然。

9. 草食と捕食で属性の使われ方が違う。
魔法拡張ゲノムのメモでは、草食は直接戦闘より防御・妨害・逃走補助への接続が自然とされる。
特に風・幻惑・遮蔽系が evasionDuration / evasionDistance / zigzagFrequency と相性が良い。
捕食は遠隔火力または補助行動として設計し、chargeDamageScale / biteDamage / meleeDamage と競合しない形にする。

10. 支配種条件では四属性の扱いが重要。
火・氷・雷・風の 4 属性魔法を扱えることが、支配種到達条件の最低条件とされている。
空間魔法は支配種のみが使える到達後能力であり、到達条件には含めない。

実装へ渡す属性決定案

A. 個体が持つべきデータ
- unlockedElements: 属性ごとの解放状態。
- elementBias: 火 / 氷 / 雷 / 風の適性重み。
- elementUseStats: 属性ごとの使用回数、成功回数、直近使用時刻。
- awakenedElements: 被弾覚醒で一時保護される属性または技能ノード。
- magicUsePriority: 魔法行動そのものを攻撃候補テーブルへ出す重み。
- fear / boredom: 自爆や連打による使用率補正。

B. 発射時の属性決定手順
1. 使用可能な属性候補を集める。
   - unlockedElements が true の属性。
   - 覚醒保護中の属性または迎撃系ノード。
   - 支配種のみ空間を候補に含める。

2. 候補ごとに基礎重みを作る。
   - elementFireBias などの属性 bias。
   - spellUnlockRate / manaControl / manaAffinity による補正。
   - magicUsePriority は「魔法を使うか」の重みであり、属性間比較には直接使いすぎない。

3. 状況補正を加える。
   - 距離が近い: 風、氷、迎撃系を上げる。
   - 複数対象: 風、雷、氷を上げる。
   - 単体高価値対象: 火、雷を上げる。
   - 逃走中: 風、氷を上げる。
   - mana field が高い: 空間や高コスト魔法の候補を上げる。ただし空間は支配種限定。

4. 感情・経験補正を加える。
   - boredom により同一属性連打を弱める。
   - fear により自爆しやすい広 AoE を弱める。
   - 成功経験がある属性は少し上げる。
   - 直近覚醒した属性は保護期間中だけ試行率を上げる。

5. 重み抽選で属性を決める。
   - 最大値固定ではなく、重み抽選にする。
   - ただし行動インデックス上の優先度は高い重み順として扱う。

6. 属性に応じて projectileSettings と payload を選ぶ。
   - 火: ダメージ寄り、heat 加算、継続ダメージ。
   - 氷: 棘生成、heat 低下、鈍足。
   - 雷: 高速、連鎖、次弾誘導、短時間 event。
   - 風: ノックバック、強制移動、vector field。
   - 空間: 転移、位置補正、mana field、支配種限定。

C. 被弾覚醒による更新手順
1. 魔法被弾イベントを受ける。
2. 被ダメージ量、最大体力比、連続被弾回数、同一属性頻度、生存継続時間から覚醒確率を計算する。
3. 受けた属性の上位相性を覚醒候補にする。
4. 覚醒したら、まず迎撃系ノードを解放する。
5. 解放直後は数世代または一定時間、ロック保護と試行率補正を付ける。
6. その後、使用実績や通常アンロックで飛翔体などへ進める。

D. 現実装との差分
現状コードでは、個体の飛翔体属性は MagicProjectileAttackAction.element の初期値 Fire に依存している。
メモ上の設計に寄せるなら、ここへ直接固定値を置くのではなく、発射直前に個体の属性選択器から MagicElement を受け取る形にする。

候補クラス案:
- MagicElementAffinityState
  - unlockedElements、elementBias、使用統計、覚醒保護を保持する。
- MagicElementSelector
  - 状況と個体状態から MagicElement を重み抽選する。
- MagicAwakeningState
  - 被弾覚醒、ロック保護、覚醒ノードを管理する。

最小実装案
1. MagicProjectileAttackAction.element 固定を残しつつ、MagicElementSelector があれば selector の結果を優先する。
2. selector がない個体は従来通り Fire を使う。
3. 最初は Fire / Ice / Lightning / Wind の unlocked と bias だけ実装する。
4. Space は支配種条件が入るまで候補外にする。
5. 被弾覚醒はイベント口だけ先に作り、確率計算とノード保護は後続で実装する。
