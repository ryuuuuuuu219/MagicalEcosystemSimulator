タスク一覧整理用

目的:
既存のタスク一覧を、実装本線と「やりたいこと」の別軸に分けて整理する。
今回の前提は、群知性や多対多戦闘より先に魔法関連システムを設計すること。
実ディレクトリの本線順は 1 → 2 → 3 → 4 → 5 → 6 → 7 に並べ替え済み。

---
基本方針

0. 遺伝子設計変更
- 現行ゲノム構造、DNA形式、世代継承処理を整理する
- 草食・捕食のゲノム差分を基準化し、後続タスクの遺伝子拡張へつなぐ

1. 本線タスクは依存順に置く。
- 1.戦闘可視化・観測基盤
- 2.知覚・地形・記憶
- 3.挙動AI分離
- 4.個体戦闘・IFF・状態遷移
- 5.魔素基盤
- 6.属性魔法
- 7.群れ意思決定・集団戦術

2. 「やりたいこと」は本線から独立した別軸とする。
- 命名は `X-n.名称`
- 横断テーマ、将来案、演出案、観測案を格納する
- 本線の依存順を壊さない

3. 群知性は戦闘だけでなく魔法とも接続する前提で後段に置く。
- 多対一 / 多対多の判断は、個体戦闘と魔法手段の両方が揃ってから設計する

4. 戦闘可視化強化は割り込みタスクとして扱う。
- フォルダ名は `1.戦闘可視化強化`
- 既存の本線タスクを 1 つ後ろへシフトし、指定個体・目標追跡カメラ、戦闘観測 UI、魔力場 / テンソル場の可視化基盤をまとめる入口として置く
- 戦闘ルール自体は 4、魔素の環境場としての仕様は 5、属性魔法の効果設計は 6 を正とし、このフォルダでは観測・カメラ・場可視化を中心に扱う

---
本線タスクの整理

1.戦闘可視化強化
- 指定個体と、その個体の現在目標を追うカメラポジション設定
- 攻撃距離、角度、クールダウン、現在目標の観測 UI
- 攻撃トレース、ダメージ表示、状態表示を同じ観測文脈にそろえる
- 魔力場、属性場、方向性を持つ場、テンソル場の最小実装と可視化

補足:
- ここでは「戦闘を観測しやすくする」ことを主目的にする
- 魔力場 / テンソル場は、4 や 5 の設計を実装へ落とすための可視化・検証入口として扱う
- heat field / threat map と重複する生成処理は増やさない

2.生態系コア強化
- 縄張り / 拠点ベース巡回の追加
- 座標好感度（場所記憶）評価の導入
- 高所・見晴らし評価の行動重み化
- threat map / heat field の行動意思決定統合
- 森と視界隠蔽の導入

補足:
- 「視覚隠蔽」は独立項目ではなく知覚基盤の一部として扱う
- ここでは「見える / 隠れる / 記憶する / 場を選ぶ」を扱い、戦闘フェーズ自体は持ち込まない

3.挙動AI分離
- 制御AIに直書きされた機能の責務分離
- 感覚（視覚/聴覚/記憶連携）・欲求・攻撃のモジュール化
- AddComponent 運用を前提にした機能境界の整理

補足:
- 2 の知覚・移動基盤を壊さずに、4 の戦闘拡張へつなぐ中間タスクとして扱う
- 「挙動の種類を増やす」より先に「挙動の責務を分ける」ことを優先する

4.戦闘システム拡張
- 肉食の待ち伏せ条件追加
- 追跡放棄条件の精緻化
- 肉食動物→肉食動物戦闘の正式化
- 摂食対象の制御をホワイトリスト式からブラックリスト式に変更
- 種族ID・IFF実装
- 高脅威判定の導入
- 撤退条件の導入
- 個体戦闘の状態遷移整理

補足:
- ここでは「誰と戦うか」「いつ交戦するか」「いつ止めるか」を固める
- 後段の群知性は、この個体戦闘ルールを共有・拡張する層として扱う

5.魔素基盤構築
- 魔素グリッドの定義
- 拡散・吸収・消費の更新式実装
- 濃度差による能力補正実装
- 生体行動と魔素場の相互作用定義
- UI観測項目の最小露出

補足:
- 群れ戦術より先に、環境場としての魔素を定義する
- 将来の群知性では「どこで戦うか」「どこへ誘導するか」の判断に魔素場を使えるようにする

6.属性魔法拡張
- 属性体系（火 / 氷 / 雷 / 風 / 空間）の効果定義
- 属性発動条件とクールダウン設計
- 戦闘 / 移動 / 環境との相互作用実装
- 観測UIで属性発動の差分可視化
- 学習・進化系への接続ポイント整理

補足:
- 群知性より前に「個体が使える手段」を増やす
- その後に、群れが魔法をどう役割分担するかを 7 で扱う

7.群知性導入
- GroupIntent データ構造定義
- 指揮者個体（リーダー）選出ロジック
- 追従 / 包囲 / 離脱判断の共有規則
- 群れ内部の意図共有プロトコル
- 戦闘フェーズとの連動
- 魔法手段を含む多対一戦闘の段階判断
- 魔法手段を含む多対多戦闘の判断基準整理

補足:
- 包囲・追い込み・囮は群れ戦術の主タスクとしてここへ集約する
- 多対多戦闘は、魔法の有無で前提が変わるため 4 と 5 の後に置く

整理時の注意

- 本線は「できる順」を優先する
- 群知性は「最後にまとめる」のではなく、「魔法込みの最終的な集団判断層」として後段へ置く
- X 系は発想メモ置き場として使い、本線の粒度を壊さない
- 7 の README は、戦闘だけでなく魔法運用との接続を書く前提にする

---
暫定結論

- 本線順は 1 → 2 → 3 → 4 → 5 → 6 → 7 に変更した
- 6 を先に詰めるのではなく、魔法関連の基盤と手段を先に設計する
- その上で群知性を「非魔法の集団戦」ではなく「魔法も含めた集団意思決定」に拡張する
