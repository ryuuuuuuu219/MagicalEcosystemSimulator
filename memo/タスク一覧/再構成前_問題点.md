# 再構成前の問題点

確認日: 2026-05-08

`進捗確認.md` に追記した方針をもとに、タスクフォルダを再構成する前に潰すべき問題を洗い出す。

この文書は 2026-05-08 時点の再構成前レビューであり、支配種仕様の現在状態は `整理後/5.支配種仕様・役割定義/README.md` を正本とする。

## 1. 現在のタスク軸と実装軸がずれている

- 旧タスクは「ロードマップ上の大項目」で切られているが、現在の実装は `organ`、`mana`、`magic`、`phase`、`generation` の実装単位で進んでいる。
- そのため、1つの実装が複数タスクにまたがっている。
  - `OrganPresetLibrary` は phase0、挙動AI分離、魔法使用、相進化にまたがる。
  - `Resource` / `ManaFieldManager` は魔素基盤、生態系コア、魔法ダメージ、世代評価にまたがる。
  - `AdvanceGenerationController` は遺伝子、世代更新、学習と進化、支配種準備にまたがる。

再構成では「実装単位」を主軸にしないと、同じ残タスクが複数フォルダへ重複する。

## 2. 完了済み・廃止予定・未接続が同じフォルダに混在している

- `2.戦闘可視化強化` は、攻撃軌跡、ダメージ数字、仮想ゲージ、場可視化がかなり実装済み。
- 一方で「目標個体と指定個体を同時に見せる戦闘カメラ」は廃止予定として追記されており、残タスクとして持つべきではない。
- `6.魔素基盤構築` は mana 基盤そのものはほぼ実装済みだが、「mana への誘引」だけが残作業として独立しそう。
- 完了済み項目を active task に残すと、未完タスクの優先度が読めなくなる。

再構成では、完了済みは `完了・保護資料` に寄せるか、README の「完了済み」に退避する必要がある。

## 3. 古いパス参照が大量に残っている

現在のコード配置は `Creatures/before` / `Creatures/after/organ` だが、タスクや設定メモには旧パスが残っている。

古い参照例:
- `Assets/script/Ingame/Creatures/Herbivore/herbivoreBehaviour.cs`
- `Assets/script/Ingame/Creatures/Predator/predatorBehaviour.cs`
- `Assets/script/Ingame/Creatures/Predator/predatorManager.cs`
- `Assets/script/Ingame/AI/ThreatMap/threatmap_calc.cs`

現在の読み替え:
- 旧 behaviour 系: `Assets/script/Ingame/Creatures/before/...`
- organ 系: `Assets/script/Ingame/Creatures/after/organ/...`
- threat map 系: `Assets/script/Ingame/Environment/Fields/ThreatMap/...`

再構成時に README を作り直すなら、旧パスをそのまま持ち越すと次の実装者が迷う。

## 4. 内部リンクと番号がシフト前のまま残っている

番号シフト後も、古いタスク番号を参照している箇所がある。

例:
- `memo/タスク一覧/10.課題ステージ/敵対勢力.txt`
- `memo/タスク一覧/0.遺伝子設計変更/現状の遺伝子構造まとめ.txt`
- `memo/タスク一覧/2.生態系コア強化`

現在は `0.phase0_organ設計` が追加され、旧 `0` 以降は +1 されている。

再構成でフォルダ名をさらに変えるなら、古い内部リンクを「移動先に合わせて更新する」か「履歴資料としてリンク切れを許容する」かを決める必要がある。

## 5. `進捗確認.md` の追記方針がまだ正式な移動仕様になっていない

追記には移動方針が書かれているが、まだ粒度が揃っていない。

現状の追記方針:
- phase0 はそのまま移動。
- 挙動AI分離は phase0 と統合。
- 遺伝子設計変更は「支配種までの遺伝子定義」と「organ構成の遺伝」に分割。
- 生態系コア強化は「魔法のダメージ判定システム」として再構成。
- 魔素基盤構築は「manaへの誘引」に縮小。
- 属性魔法拡張は「魔法の使用」と「魔法の資質の遺伝」に分割。
- 学習と進化は「世代更新と遺伝」として再構成。
- 支配種到達条件の前工程として「相の進化」を追加。
- 環境外乱は `12.課題ステージ（環境外乱）` として再構成。

問題:
- 分割後の番号順が未確定。
- 元フォルダ内のどのファイルをどちらへ移すか未確定。
- 「移動」「統合」「完了扱い」「保護資料」の基準が未確定。

## 6. `organ` と旧 behaviour の責務が未固定

コード上は `OrganPresetLibrary.EnsureHerbivore()` / `EnsurePredator()` で organ を付与している。

ただし、本編制御はまだ旧 behaviour の `Update()` / `FixedUpdate()` / `ComputeTotalVector()` が中心。

問題:
- `AnimalBrain.TickBrain()` を本流にするのか。
- 旧 behaviour は bootstrap / legacy fallback として残すのか。
- organ の `Desire` / `Action` をどこまで実行責務にするのか。

ここが曖昧なままだと、phase0、挙動AI分離、魔法使用、mana誘引の全タスクが重複する。

## 7. 魔法ダメージ、属性魔法、mana誘引の境界が曖昧

関連コード:
- `MagicProjectile`
- `Magic2FieldManager`
- `field2AI`
- `ManaFieldManager`
- `ManaFieldAttractionDesire`
- `MagicProjectileAttackAction`

問題:
- 「魔法のダメージ判定システム」は、属性魔法タスクなのか、生態系コアタスクなのか、戦闘システムタスクなのか曖昧。
- `field2AI` は場から damage / hazard を計算できるが、主移動や魔法行動へ接続されきっていない。
- `ManaFieldAttractionDesire` はあるが、旧 behaviour の移動合成にはまだ薄い。

再構成では、少なくとも次の3つを分けた方がよい。
- 魔法が当たったときの damage / field 反映。
- mana field へ寄る、避ける、吸う行動。
- 魔法を使うかどうかのAI判断。

## 8. 遺伝・世代更新・魔法資質・organ構成の遺伝が絡まりやすい

関連コード:
- `HerbivoreGenome`
- `PredatorGenome`
- `GenomeSerializer`
- `AdvanceGenerationController`
- `OrganPresetLibrary`
- `AIComponentGene`

問題:
- 現在の genome と organ 構成の関係がまだ弱い。
- `AIComponentGene` はあるが、世代更新や DNA 表現へ統合されているとは言い切れない。
- 魔法資質、相進化、支配種までの遺伝子定義を同じフォルダに入れると大きすぎる。

再構成では、次を分けると見通しがよい。
- 世代更新とDNA入出力。
- 支配種までの遺伝子定義。
- organ構成の遺伝。
- 魔法資質の遺伝。

## 9. 支配種と相進化の関係

2026-05-11 の phase5 実装で、支配種と相進化の責務は整理済み。

現在の整理:
- `predator -> highpredator` は相進化。
- `highpredator -> dominant` は相そのものの突然変異ではなく、通常属性魔法二種以上による支配種化。
- 支配種化は `DominantAscensionAction` が担当する。
- `PredatorPhaseEvolutionAction` と旧 `predatorBehaviour.TryPhaseEvolution()` は、どちらも `predator -> highpredator` までに制限済み。

現在の正本は `整理後/4.相の進化/README.md` と `整理後/5.支配種仕様・役割定義/README.md`。

## 10. 課題ステージと環境外乱はUI入口だけで、実行ロジックが薄い

関連コード:
- `WorldUIManager.MenuFlow`
- `WorldUIManager.ButtonBindings`
- `WorldUIManager.PhasePopulation`
- `GenerationLog`
- `PerformanceBudgetMonitor`

問題:
- Disturbance UI はあるが、外乱パラメータを期間イベントとして適用する controller が見当たらない。
- 課題ステージの勝敗、報酬、敵対勢力、クリア条件が実行時ロジックとして独立していない。
- `11.課題ステージ` と `12.課題ステージ（環境外乱）` は分けてよいが、共通の stage controller が必要。

再構成後も、ここは低進捗として明確に残すべき。

## 11. 保護資料にも古い出力パスが残っている

`26.5.6_魔素移行資料` は `memo/設定` へ移動済みだが、資料内には旧パスが残る。

例:
- `memo/タスク一覧/26.5.6_mana_simulation_result.csv`
- `memo/タスク一覧/26.5.6_魔素数値暴走シミュレーション.py`

保護資料なので中身を勝手に直すべきかは微妙。

方針候補:
- 保護資料は履歴として中身を変えない。
- 代わりに `memo/設定/26.5.6_魔素移行資料/README.md` を作り、現配置と旧記述の読み替えを書く。

## 12. X系メモの扱いが未決定

X系は将来案として有用だが、active task に混ぜると優先順位が濁る。

問題:
- 戦術演出、判断ログ、陣形、縄張りは、群知性や戦闘AIに接続できるが現時点では未実装。
- 遺伝子拡張候補は、`支配種までの遺伝子定義` や `魔法資質の遺伝` と重複する可能性がある。
- 熱量場関連は `魔法のダメージ判定システム` や `manaへの誘引` と近い。

再構成では、X系は `保留・将来案` として残し、active numbered task へ必要な項目だけ転記するのがよい。

## 13. 進捗率に基準がない

`進捗確認.md` の進捗率は現状把握としては使えるが、再構成後の完了判定には弱い。

問題:
- 何ができたら 100% なのかがフォルダごとに違う。
- 実装済み、設計済み、UI入口あり、実行ロジックあり、検証済みが混ざっている。

再構成後の各フォルダには、最低限以下を置く必要がある。
- 現状
- 残タスク
- 対象スクリプト
- 完了条件
- 旧フォルダから移した資料

## 推奨する再構成前の決定事項

1. `整理後` フォルダに複製してから再構成するか、既存フォルダを直接リネーム・移動するか。
2. 完了扱いにするフォルダを決める。
   - 候補: `2.戦闘可視化強化`
3. 統合するフォルダを決める。
   - 候補: `4.挙動AI分離` -> `0.phase0_organ設計`
4. 分割するフォルダを決める。
   - `1.遺伝子設計変更`
   - `7.属性魔法拡張`
   - `9.学習と進化`
5. 新設するフォルダ名と番号順を決める。
   - `相の進化`
   - `魔法のダメージ判定システム`
   - `manaへの誘引`
   - `魔法の使用`
   - `魔法の資質の遺伝`
   - `世代更新と遺伝`
   - `支配種までの遺伝子定義`
   - `organ構成の遺伝`
6. 保護資料とX系メモは、active task から分離する。
