目的

- 実装時に herbivore / predator / 共通AI で解釈がぶれないようにする。

対象資料

- memo/タスク一覧/1.生態系コア強化/設定：ゲノム構造（基礎拡張）.txt

作業項目
1. 既存仕様の抜け確認

2. 意思決定レイヤーの定義

3. ベクトル合成ルールの定義
- attraction / repulsion / boundary / terrain 補正の合成順を決める。

4. ログ観測項目の定義

5. IFF / 警戒状態の仕様化

- 警告後に離脱なら警戒へ戻す、接近なら戦闘状態へ移すなどの分岐を明記する。

6. 初期縄張りルールの仕様化
- 草原を中心とした円状領域を基礎実装の縄張りとして定義する。
- 縄張り侵入をターゲット候補化する条件を整理する。
- 現状の「一定距離内の個体を敵性判断するのみ」から、どこまでを 1 段階目で置き換えるか決める。

現在の整理状況
- 設計確定は中程度まで前進している。
- 場データの概念整理と演算仕様の土台はかなり固まっている。

決定済み事項
- 現在の移動スクリプトは `food / prey`、`threat`、`boundary`、`wander` の重み付き合成で動いている。
- 草食は近接回避 + zigzag evasion を持つ。
- 肉食は proportional navigation 補正つき追跡を持つ。
- 実移動は `AnimalAICommon.ApplyMovement()` により、慣性つき・回頭制限つき・地形追従モデルで処理されている。
- heat field は `Qbody / Qenv` を内部表現とし、温度換算、熱交換、分解熱、恒星熱、移動評価、状態異常まで演算仕様が定義済みである。
- mana field も `Mbody / Menv`、吸収、消費、拡散、死骸分解、転移、移動評価まで演算仕様が定義済みである。
- `設定：field設計まとめ.txt` は要点のみを持ち、heat / mana の詳細は各演算シミュレーション文書を正として参照する構成に整理済みである。

状態名ごとの発火条件 / 停止条件

現状整理
- 現状コードでは、探索 / 回避 / 追跡 / 離脱の多くが明示 enum ではなく、距離・記憶・重み・一時フラグで暗黙に切り替わっている。
- 草食は `ComputeEvasionVector()` の `evasionW > 0` が最優先で、発火すると他の移動要求より先に回避を返す。
- 草食の通常回避は `wThreat > max(0.1, genome.escapeThreshold)` で `food` と `wander` を抑制する。
- 草食の摂食停止は `isFood == true` で `Vector3.zero` を返す形になっている。
- 肉食の追跡は `GetBestRememberedPrey()` で獲物候補が見つかったときに成立する。
- 肉食の攻撃は `TryCombatActions()` が成功した瞬間に成立する。
- 肉食の通常離脱は専用状態としては未分離で、`wThreat > 0.5` 時に `prey` と `wander` を抑制する形で実質表現されている。
- 警戒 / 警告 / 戦闘状態の名前付き状態は、現状は仕様上のみで、共通の状態管理としては未実装。

推奨整理

- 探索
  発火条件:
  `threatWeight < escapeThreshold`
  かつ有効な `food / prey` ターゲットがない、または誘引重みが小さいとき。
  停止条件:
  `foodWeight > searchExitFoodThreshold`
  または `preyWeight > searchExitPreyThreshold`
  または `threatWeight >= escapeThreshold`
  または `heatWeight >= heatAvoidThreshold`

- 回避
  発火条件:
  `threatWeight >= escapeThreshold`
  または近接危険回避の `evasionW > 0`
  または `heatWeight >= heatAvoidThreshold`
  停止条件:
  `threatWeight < escapeThreshold - escapeHysteresis`
  かつ `evasionTimer <= 0`
  かつ `heatWeight < heatAvoidThreshold - heatHysteresis`
  備考:
  近接回避と通常回避は強度が違うので、仕様上は「近接回避」と「広域回避」を分けて書いてもよい。

- 追跡
  発火条件:
  有効な `currentTarget` または `GetBestRememberedPrey()` が存在し、
  `preyWeight > chaseEnterThreshold`
  かつ `threatWeight < chaseBlockThreatThreshold`
  のとき。
  停止条件:
  ターゲット消失、
  `memoryPrey` 失効、
  `threatWeight >= chaseBlockThreatThreshold`、
  または攻撃可能距離に入り `戦闘` へ移行したとき。

- 離脱
  発火条件:
  追跡中または警戒中に `threatWeight >= disengageThreatThreshold`
  もしくは IFF 上の劣勢判定、
  もしくは縄張り / 境界 / heat の複合危険が閾値を超えたとき。
  停止条件:
  `threatWeight < disengageThreatThreshold - disengageHysteresis`
  かつ追跡対象との距離が十分離れ、
  再追跡条件を満たさないとき。
  備考:
  現状コードでは未分離なので、まずは `回避` の上位概念ではなく「追跡や警戒から抜ける遷移名」として定義するのが安全。

- 警戒
  発火条件:
  直接視認、攻撃痕跡、`threat map` 高値、被攻撃イベント、縄張り侵入のいずれかを検知したとき。
  停止条件:
  一定時間新規刺激なし、
  `threat map` 低下、
  対象不明のまま `alertTimeout` 経過、
  または追跡 / 戦闘 / 離脱へ遷移したとき。
  備考:
  `threat map` はここでは状態遷移の補助根拠としては使ってよいが、単独決定因にはしない方が安全。

- 警告
  発火条件:
  警戒対象が識別済みで、
  IFF 上は即戦闘ではなく、
  かつ距離・相対速度・侵入条件が警告レンジに入ったとき。
  停止条件:
  対象が離脱した、
  応答がなく接近継続した、
  または敵対確定で `戦闘` へ移行したとき。
  備考:
  現状未実装なので、後からでも追加しやすいように IFF / 距離 / 継続時間だけで書き始めるのがよい。

- 戦闘
  発火条件:
  追跡または警告中の対象が攻撃条件を満たし、
  `TryCombatActions()` 相当の攻撃判定が成立したとき。
  停止条件:
  対象死亡、
  対象喪失、
  `attackCooldown` 中の攻撃失敗継続、
  エネルギー不足、
  または `離脱` 条件成立時。
  備考:
  現状の実装では `CombatState` はクールダウン管理中心で、戦闘そのものの状態 enum ではない。

仕様化の優先順推奨
- まず `探索 / 回避 / 追跡 / 戦闘` の 4 状態を先に固定する。
- 次に `離脱` を「追跡・警戒から安全側へ抜ける状態」として追加する。
- `警戒 / 警告` は IFF と縄張り仕様が固まる段階で追加する。
- 変数名は既存実装に寄せて、`escapeThreshold`、`threatWeight`、`preyWeight`、`foodWeight`、`memorytime`、`attackCooldown` 系を優先して使う。

- 実装側が参照する優先順位と停止条件が明記されている。
